Module 2

Performance & Parallelism

--checkers --builders --singleThreaded

Module 1 established that TypeScript 7 is the same language on a faster engine. This module explains where the speed comes from — and gives you the controls to tune it. Open each idea.

native speed Half the win: no JavaScript runtime overhead +

The old compiler was itself a TypeScript program compiled to JavaScript, so it ran on Node.js and inherited the JavaScript engine's just-in-time warm-up and startup costs. The native compiler runs as a compiled binary instead. Roughly half of the overall ~10× improvement comes from this alone: native code execution with no JIT overhead and no Node.js startup tax.

This part of the speed-up is automatic. You get it simply by running the native compiler — there is nothing to configure. The other half is parallelism, which is where the controls in this module come in.

parallelism The other half: doing many things at once +

The single-threaded JavaScript runtime had to do the compiler's work one step at a time. Go made it practical to run parts of the build concurrently. TypeScript 7 now parallelises parsing, type-checking, and emitting.

Parsing and emitting are the easy wins: each file can largely be handled independently, so this scales well with codebase size at low overhead. Type-checking is harder — files share type information from their dependencies and the global scope, so the checkers can't run in complete isolation without wasting computation and memory. TypeScript 7 splits type-checking across a configurable number of worker processes (called "checkers"), each with its own view of the program, dividing the work while still producing the same answers.

10× claim How to read the speed numbers honestly +

The headline figure — "often about 10× faster than TypeScript 6.0" — is Microsoft's own measurement on Microsoft's own hardware and codebases. It is representative, not a guarantee. Your speed-up depends on your codebase shape, your machine's core count and memory, cache warmth, and how much of your build is type-checking versus emit.

A realistic expectation: large codebases on many-core machines see the biggest gains; small projects see less dramatic numbers because there is less work to parallelise and the fixed overhead is proportionally larger. The honest way to know your number is to measure it on your own project — which is exactly what the Benchmark lab tab walks you through.

Refresh your knowledge?

Concurrency vs. parallelism. These words are often used loosely, but the distinction matters here. Concurrency is structuring a program so multiple tasks are in progress over the same period — they may be interleaved on a single worker. Parallelism is multiple tasks literally running at the same instant on multiple workers. TypeScript 7 does real parallelism: separate workers crunching different parts of your program simultaneously.

What a "thread" or "worker" is. A thread is an independent stream of execution within a program. Multiple threads can run on multiple CPU cores at once. The old JavaScript-based compiler was effectively single-threaded for this work: Node.js worker threads exist, but they communicate by copying serialized messages between isolated engine instances, which is too costly for the tightly-shared data a type-checker needs. Go's lightweight threads (goroutines) share the same memory directly, so several of them can read the same symbol table at once without copying. That shared-memory model is what made parallel type-checking practical — and it's why the port chose Go.

Why more workers isn't always faster. Splitting work across workers has a cost: coordinating them, and giving each its own slice of memory. Past a certain point — especially on a small project or a machine with few cores — adding workers stops helping and can even slow things slightly, because the coordination overhead outweighs the work saved. This is why the defaults are modest and why benchmarking your own project beats assuming "more is better."

Accurate as of June 2026 The flag names and defaults below track the TypeScript 7.0 RC. Preview details can shift — if a flag behaves differently, check the official TypeScript site and the @typescript/native-preview npm page.

The three parallelism flags at a glance

Native code speed is automatic. Parallelism is the part you can tune, and the RC exposes three flags to control it. The defaults are sensible for most development machines; you mainly reach for these when tuning a CI runner with a known core count and memory ceiling.

FlagDefaultWhat it does
--checkers4Number of parallel type-checking workers.
--builders—Number of project-reference builds run in parallel (monorepos).
--singleThreadedoffDisables parallelism entirely (one thread for everything).

--checkers — type-checking workers

This sets how many type-checking workers run at once. The default is 4, which balances speed against memory on a typical development machine. Raising it can speed up larger codebases on machines with more cores — at the cost of proportionally higher memory use. On a constrained CI runner with few cores, lowering it avoids overhead that would otherwise slow you down.

# more workers for a big codebase on a beefy machine
npx tsc --noEmit --checkers 8

# fewer workers for a small / low-core CI runner
npx tsc --noEmit --checkers 2
Determinism caveat. In rare cases, varying the number of --checkers can surface order-dependent results. If your team wants byte-for-byte identical output everywhere, fix --checkers to the same number across all build environments. (This relates to the stableTypeOrdering setting covered in Module 3.)

--builders — parallel project-reference builds

A monorepo often splits into many smaller TypeScript projects wired together with project references. TypeScript 7 can build several of those projects at once, and --builders sets how many run in parallel. Because the project dependency graph is declared explicitly in your tsconfig.json, partitioning the work is straightforward — so unlike --checkers, varying --builders does not change your results.

# build a monorepo, several projects at a time
npx tsc --build --builders 4
It multiplies with --checkers. Each builder spins up its own checkers, so the two compound. --checkers 4 --builders 4 can launch up to 16 type-checkers at once — appropriate on a many-core build server, but excessive on a standard laptop, where it can exhaust memory. Set both deliberately for your hardware.

--singleThreaded — turn parallelism off

This caps type-checking to a single worker and also forces parsing and emitting onto one thread. You give up the parallel half of the speed-up (you keep the native-code half). It's not for everyday use — it's a deliberate tool for specific situations: debugging a confusing result, making a clean apples-to-apples comparison against TypeScript 6.0, orchestrating parallelism yourself from the outside, or running where memory is very tight.

# deterministic, low-memory, single-threaded run
npx tsc --noEmit --singleThreaded

The rebuilt --watch mode

Watch mode — recompiling automatically when files change — was also rebuilt. The old approach leaned on polling, which became expensive on large projects with big node_modules directories. TypeScript 7's watcher is built on a Go port of the Parcel bundler's file-watcher (the same lineage that underpins file-watching in VS Code), giving more resource-efficient watching across platforms.

# recompile on save, efficiently
npx tsc -w
Refresh your knowledge?

What a CPU core is. A modern processor has multiple cores — independent units that each execute instructions. Four cores can genuinely do four things at the same instant. (With "hyper-threading"/SMT, the operating system may report more logical cores than physical ones; for our purposes, think in terms of how many independent streams of work your machine can truly run at once.) When TypeScript 7 starts 4 checker workers, it's aiming to keep up to 4 cores busy.

How more workers help — up to a point. If you have 8 idle cores and a big type-checking job, spreading it across 8 workers can finish far faster than one worker doing it all. But two things cap the benefit. First, you can't use more cores than you physically have — 16 workers on a 4-core laptop just take turns, adding coordination cost for no gain. Second, each worker uses memory; on a machine that's short on RAM, too many workers cause swapping that's slower than running fewer. The practical rule: match worker count to the machine, and measure rather than guess.

Finding your machine's core count. It helps to know what you're working with before tuning --checkers:

# macOS / Linux
nproc                 # Linux: prints logical core count
sysctl -n hw.ncpu     # macOS: logical cores

# cross-platform, via Node
node -e "console.log(require('os').cpus().length)"

A common starting point is to leave --checkers at its default of 4 and only adjust once a benchmark shows it helps.

Goal

Measure the speed-up on a real codebase yourself, then see how --checkers affects it. You'll end with your own before/after number — the only number that actually describes your project. Use a medium-or-larger TypeScript project you already have (the more files, the clearer the effect).

Step 1 — Install both compilers side by side

You need TypeScript 6.0 (the baseline) and the native preview together. This is the side-by-side setup from Module 1.

npm install -D typescript @typescript/native-preview

Step 2 — Get a baseline with TypeScript 6.0

Run a type-check-only pass (no emit, so we isolate checking speed) and ask the compiler for its own diagnostics, which include a total time:

# classic compiler, by its package bin path to avoid name clashes
node ./node_modules/typescript/bin/tsc --noEmit --extendedDiagnostics

Note the Total time line near the bottom. Run it twice and use the second run — the first warms the filesystem cache, and you want a warm-cache number for a fair comparison.

Step 3 — Run the native compiler

Now the same check with the native preview. (On the RC the binary is tsc; on older nightlies it's tsgo.)

npx tsc --noEmit --extendedDiagnostics

Compare the Total time to your baseline. Divide the 6.0 time by the 7.0 time to get your speed-up ratio. Don't be surprised if it differs from "10×" in either direction — that figure was measured on a very large codebase.

Step 4 — Sweep --checkers

See how worker count changes things on your project and your machine. Run each and record the time:

npx tsc --noEmit --extendedDiagnostics --checkers 1
npx tsc --noEmit --extendedDiagnostics --checkers 2
npx tsc --noEmit --extendedDiagnostics --checkers 4   # default
npx tsc --noEmit --extendedDiagnostics --checkers 8

Fill in a small table as you go:

RunTotal timevs. 6.0 baseline

What you'll likely see. Time drops sharply from 6.0 to 7.0 even at --checkers 1 (that's the native-code half of the win). Adding checkers helps up to a point, then flattens — and on a smaller project or a low-core machine it may not improve much beyond the default, or may even tick upward. That plateau is the lesson: parallelism has diminishing returns, and the right number is the one your own numbers show.

Step 5 (monorepo, optional) — try --builders

If your project uses project references, build it with parallel builders and watch the multiplication effect:

npx tsc --build --builders 2
npx tsc --build --builders 4

Remember builders compound with checkers — keep an eye on memory, and don't push both high on a laptop.

Refresh your knowledge?

How to benchmark fairly. A "speed test" is only meaningful if the comparison is clean. A few rules that keep your numbers honest:

Warm vs. cold cache. The first time you run a compile, the operating system reads your files from disk for the first time; the second time, those files are cached in memory and reads are far faster. This is "cold" vs. "warm" cache. Always discard the first run and compare warm-cache runs against each other, or you'll be measuring disk speed, not compiler speed.

Wall-clock vs. CPU time. Wall-clock time is how long you waited (a stopwatch). CPU time is the total processor work, summed across all cores — so a parallel run can use more CPU time while finishing in less wall-clock time. For "how much faster does it feel," wall-clock is what matters; the compiler's Total time reports the wall-clock duration.

Hold everything else still. Close other heavy apps, stay on power (laptops throttle on battery), don't change the source between runs, and run several times to average out noise. Change one variable at a time — that's why the sweep above varies only --checkers.

Why your number won't match the headline. Microsoft's "~10×" came from a specific, very large codebase on specific hardware. Smaller projects have less work to parallelise and proportionally more fixed overhead, so they show smaller multiples. A 3–5× result on a mid-size project is a perfectly normal, honest outcome — it doesn't mean you did anything wrong.

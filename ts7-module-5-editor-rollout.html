Module 5

Editor / Language Service & Rollout

language service LSP adoption

The fast compiler powers more than your builds — it powers your editor. This final module covers the editor side of the native port and how to actually roll TypeScript 7 out to a team without drama. Open each idea.

language service The same engine drives your editor +

The "language service" is the part of TypeScript that powers editor features: completions, hover tooltips, go-to-definition, error squiggles, refactorings. It runs on the same foundation as the command-line compiler, so the native port's speed shows up in the editor too — teams report roughly 8–10× faster project-load and type-check times in the editor on large repositories.

You try it through the TypeScript Native Preview extension for VS Code, updated frequently and already in widespread use. It respects most of the same settings and features as the built-in TypeScript extension, so for most people it feels like their familiar editor — only faster to wake up and respond on a big codebase.

LSP Built on the Language Server Protocol +

The older editor integration spoke a TypeScript-specific protocol ("TSServer") that predates today's standards. The native port instead speaks the Language Server Protocol (LSP) — the editor-agnostic standard most modern languages use. The practical pay-off: the same language service can drive not just VS Code but most modern editors, and even tools like Copilot CLI, rather than being locked to one editor.

Because the protocol changed, some behaviour specific to the old VS Code extension may differ, and TypeScript server plugins that hooked into the old protocol don't work under the native preview (this is expected). The upside is bigger than the cost: LSP is why "TypeScript 7 in your editor" isn't a VS-Code-only story.

parity What's solid vs. still catching up +

The language service was more heavily rewritten than the type-checker, so it's worth knowing where it stands. Working and ready for daily use: completions, hovers (including expandable hovers), go-to-definition and go-to-source-definition, diagnostics, inlay hints, code lenses, signature help, and — after earlier gaps — auto-imports, find-all-references, and rename, including in projects with project references.

Still catching up: a few niceties like semantics-enhanced highlighting and some specialised import-management commands, plus the server-plugin story noted above. The headline: the editor experience is described by the team as powerful, stable, and fast today — the remaining gaps are refinements, not fundamentals. What a language server is. When you hover a variable and see its type, or press "go to definition" and jump across files, an editor isn't doing that analysis itself — it's asking a language server: a background program that understands your code and answers questions about it. The editor sends requests ("what's at line 12, column 7?") and the server replies ("a string named title, defined in util.ts"). Separating the two means the heavy language smarts live in one place and any editor can tap them.

What LSP is, and why it matters. The Language Server Protocol is a shared standard for how editors and language servers talk to each other. Before it, every language needed custom integration code for every editor — an N×M problem. With LSP, a language server speaks one protocol and works in every LSP-capable editor; an editor speaks one protocol and supports every LSP language. TypeScript 7 adopting LSP is what lets its fast language service reach beyond VS Code to other editors with little extra work.

Editor features as "questions to the server." It helps to see the familiar features as request types: completion (what can I type here?), hover (what is this?), definition (where is this declared?), references (where is this used?), rename (change this name everywhere safely), diagnostics (what's wrong?). Each is a standard LSP message. When the docs say a feature is "ported," they mean the native server now answers that request correctly.

Accurate as of June 2026 Rollout advice here reflects the TypeScript 7.0 RC, where production emit and a stable API are still settling. As 7.0 reaches stable GA, some "wait for stable" cautions below will relax — re-read the official release notes when you plan a real cutover.

The guiding principle: separate "check" from "ship"

The safest adoptions exploit one fact established across this course: TypeScript 7's type-checking is essentially complete and matches 6.0, while a few emit-side features may still be settling during the preview. So you can adopt 7.0 for the fast feedback (type-checking, editor) well before you depend on it for production JavaScript emit. Most of the playbook follows from that split.

Step 1 — Pilot in CI first (lowest risk)

Add a 7.0 type-check job to CI alongside your existing 6.0 build — not replacing it. It runs tsc --noEmit on every push and surfaces any divergence without affecting what you ship. This is the cheapest way to gain confidence on real code.

# CI: run both, compare. 7.0 job is informational at first.
# 6.0 (authoritative for now)
node ./node_modules/typescript/bin/tsc --noEmit
# 7.0 (fast, informational)
npx tsc --noEmit

Step 2 — Let developers opt in for editing

Have a few developers install the TypeScript Native Preview extension and use it day-to-day. The editor is where the speed is most felt, and it's low-risk because it doesn't change what's committed or shipped — only how fast the editor responds. Collect their feedback on any behaviour differences (recall a few LSP-related changes and the server-plugin gap from the Overview).

Step 3 — Validate side-by-side until results agree

Use the techniques from Modules 2–4: run 6.0 and 7.0 on the same code and confirm they report the same errors. Reach the "clean and honest build" bar from Module 3 (6.0 clean with stableTypeOrdering, no ignoreDeprecations). When 7.0 consistently agrees with 6.0 across your CI history, it has earned trust.

Step 4 — Flip the default for type-checking

Make 7.0 the authoritative type-check in CI and the default local check script. Keep 6.0 reachable (via the tsc6 alias from Module 4) for tooling that still needs its API and as a fallback. At this point your team gets the fast feedback loop everywhere, while production emit can still go through whatever you trust.

Step 5 — Stage a monorepo, project by project

In a monorepo, don't flip everything at once. Adopt leaf projects (few dependents) first, watch them through a release cycle, then move up the dependency graph. Use --builders (Module 2) once multiple projects are on 7.0 to parallelise the whole build. Gate any production-emit cutover on the stable GA release and your own test suite passing.

Rule of thumb. Adopt fast feedback (editor, CI type-check) eagerly; adopt production emit conservatively. The first is reversible and low-stakes; the second is what actually ships to users.

What CI (continuous integration) is. A CI system automatically runs your checks — type-check, tests, lint — every time code is pushed, on a clean server rather than someone's laptop. It's the natural first place to pilot a new compiler because it's isolated (a failing experimental job needn't block anyone), reproducible (clean environment every run), and visible (everyone sees the results). Adding a 7.0 job next to your 6.0 job means real code gets checked by the new compiler continuously, with zero risk to what ships.

Why "informational" jobs are the safe on-ramp. A CI job can be marked non-blocking (it reports results but doesn't fail the build). Running the 7.0 check this way lets you watch it against months of real commits and compare it to the trusted 6.0 job before you ever make it authoritative. Only once it consistently agrees do you promote it to blocking. This is the same parallel-change discipline as the package aliases in Module 4: introduce the new thing beside the old, prove equivalence, then switch.

Reversibility as the deciding factor. A good rule for any migration: adopt the reversible things first. Switching your editor or an informational CI check to 7.0 can be undone in seconds and affects nobody downstream. Switching production emit ships bytes to users and is far harder to walk back. Sequencing adoption from most- to least-reversible is what keeps a rollout calm.

Goal

Put the whole course together. Take a mid-size project (ideally a small monorepo) end to end: a clean 7.0 build, tuned parallelism, coexistence wired for tooling, the editor on the native preview, and a before/after benchmark to show for it. This is the exercise that earns the certification.

Step 1 — Honest 6.0 baseline (Module 3)

Get the project building cleanly under 6.0 with no deprecated flags, then confirm the honest-build bar:

node ./node_modules/typescript/bin/tsc --noEmit --stableTypeOrdering

Fix any deprecation or strict errors first — this is where most migration effort actually lives.

Step 2 — Install and coexist (Modules 1 & 4)

Bring in 7.0 and wire the aliases so builds run 7.0 while tooling keeps the 6.0 API:

{
  "devDependencies": {
    "typescript": "npm:@typescript/typescript6@^6.0.0",
    "typescript-7": "npm:typescript@rc"
  },
  "scripts": {
    "check": "tsc --noEmit",
    "check:6": "tsc6 --noEmit",
    "lint": "eslint ."
  }
}
rm -rf node_modules package-lock.json && npm install
npx tsc --version    # 7.0
npx tsc6 --version   # 6.0

Step 3 — Confirm 7.0 agrees, and benchmark (Module 2)

# they should report the same errors
npm run check:6
npm run check

# benchmark (warm cache; second run)
npx tsc --noEmit --extendedDiagnostics --checkers 4

Record the before (6.0) and after (7.0) Total time, and your speed-up ratio. For a monorepo, build in parallel and note the effect:

npx tsc --build --builders 4

Step 4 — Editor on the native preview (Module 5)

Install the TypeScript Native Preview extension, enable it for the workspace, and confirm the editor essentials work on your code: completions, hover, go-to-definition, find-all-references, rename, auto-imports. Note anything that behaves differently (server plugins, any still-catching-up feature).

Step 5 — Produce a migration report

Write up a short rollout report — the same artifact you'd hand a team lead. Include:

  • The before/after benchmark numbers and your --checkers sweep.
  • Confirmation that 6.0 and 7.0 agree on errors (the honest-build check).
  • Any config changes the migration required (dropped flags, explicit defaults).
  • The coexistence setup (aliases) and which tooling depends on the 6.0 API.
  • Editor findings and any feature gaps relevant to your team.
  • A recommendation: what to adopt now (editor, CI type-check) vs. what to gate on stable GA (production emit).

What a migration checklist buys you. A migration is a sequence of small, verifiable changes — not a single leap. Writing the steps down as a checklist turns an intimidating "upgrade everything" into a series of done/not-done items, each independently checkable. It also makes the work reviewable by someone else and repeatable across other projects. Every step in this capstone maps to one earlier module, which is the point: the checklist is the course, applied.

What goes in a good rollout report. A useful report answers three questions for a decision-maker: Does it work? (the honest-build check — 6.0 and 7.0 agree), Is it worth it? (the benchmark — your real speed-up, not the headline number), and What's the risk? (what's adopted reversibly now vs. gated on stable GA). Concrete numbers and an explicit now-vs-later recommendation are what turn "we tried the new compiler" into a decision a team can act on.

Why "reversible now, risky later" is the through-line. Across the whole course, the safe pattern repeats: run the new beside the old, prove they agree, then switch — editor and CI type-check first (reversible, low-stakes), production emit last (gated on GA and your tests). If you remember one thing past the exam, let it be that sequencing. It's what makes adopting a brand-new compiler a calm, staged process rather than a gamble.

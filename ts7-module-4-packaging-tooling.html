Module 4

Packaging, Tooling & Coexistence

@typescript/typescript6 npm aliases 7.1 API

The compiler is fast and your build is clean — but your project isn't just a compiler. It's a linter, a formatter, editor plugins, and a web of dependencies that all expect "TypeScript" to mean the package they've always known. This module is about making the new compiler and the existing ecosystem coexist gracefully. Open each idea.

package journey From @typescript/native-preview to typescript +

During the preview, the native compiler lives in its own package, @typescript/native-preview, deliberately separate so it can't collide with the typescript package you already have. The binary's name has moved with the phase: nightlies use tsgo, while the 7.0 RC ships a tsc binary.

At the stable release, the native compiler takes over the regular typescript package and the familiar tsc command — at which point the preview package retires. So the "journey" is: experiment now under the preview name, and the day stable lands, your typescript dependency simply becomes the fast one. Knowing where you are on that path tells you which package name and binary to expect.

compat package @typescript/typescript6 keeps the old API available +

Here's the tension. The new tsc wants the typescript name. But lots of tooling imports the typescript package to call its API. If typescript suddenly means the new compiler — whose API isn't stable yet — that tooling breaks.

The bridge is a compatibility package, @typescript/typescript6. It re-exports the stable TypeScript 6.0 programmatic API and ships an executable named tsc6, so it can sit alongside 7.0's tsc without a name clash. The pattern: point your tooling at the 6.0 API through this package, while your builds use the fast 7.0 compiler. The Tooling and Coexistence-lab tabs show exactly how.

7.1 API Why the compat package is necessary right now +

You might ask: why not just have the tooling use the new compiler's API? Because it doesn't exist in stable form yet. TypeScript 7.0 deliberately ships without a stable programmatic API — the interface that tools like typescript-eslint, ts-morph, and custom transformers import. That stable API is targeted for TypeScript 7.1, at least several months after 7.0.

So the compat package isn't a permanent fixture — it's a bridge across a specific gap in time. Once 7.1 ships a stable API and the ecosystem updates to it, the aliasing dance in this module becomes unnecessary. Until then, it's how you get 7.0's speed without breaking your linter.

Refresh your knowledge?

What a peer dependency is. Most dependencies are things a package bundles for itself. A peer dependency is different: it's a package the tool expects you to provide, and that it will share with you rather than install its own copy. typescript-eslint declares typescript as a peer dependency — it doesn't carry its own TypeScript; it uses whatever typescript is installed in your project. That's exactly why the package name matters so much here: the tool reaches for "the typescript package," and whatever that name resolves to is what it gets.

How npm version resolution works (briefly). When several packages each want typescript, npm tries to satisfy them from a single shared copy when their version ranges overlap, and otherwise installs more than one copy nested in the tree. Two different TypeScript versions sneaking into one project is a classic source of confusing, contradictory type errors — the compiler you run and the one your tooling loaded aren't the same. Keeping the resolution deliberate (which the aliases in this module do) avoids that.

What "re-export the API" means. The compat package doesn't reimplement TypeScript 6.0 — it re-exports it: it forwards the real 6.0 API under a package name of your choosing. So when a tool imports from typescript and that name is aliased to the compat package, the tool transparently gets the genuine 6.0 API it was written against, even though your tsc command is running 7.0.

Accurate as of June 2026 The package names, alias syntax, and the typescript@rc tag below track the TypeScript 7.0 RC. These details are actively changing as 7.0 moves to stable and 7.1 approaches — confirm against the official 7.0 release notes and the @typescript/typescript6 npm page before relying on a specific command.

The problem, concretely

Say your project uses typescript-eslint. It imports the typescript package to understand your code. You also want 7.0's fast tsc for builds. If you simply install the 7.0 preview as typescript, the linter loads an API that isn't stable yet and misbehaves. You need typescript (the name your tooling reads) to mean 6.0's API, while your build command runs 7.0. npm aliases make one package name point wherever you choose.

The recommended alias

Point the typescript name at the compat package so all tooling that peer-depends on typescript transparently gets the 6.0 API:

# installs the compat package under the name "typescript"
npm install -D typescript@npm:@typescript/typescript6

Equivalently, in package.json:

{
  "devDependencies": {
    "typescript": "npm:@typescript/typescript6@^6.0.0"
  }
}
But there's a catch. Doing only this leaves you with just a tsc6 executable — you've given the typescript name to the 6.0-API compat package, so there's no 7.0 tsc anymore. Tooling is happy, but your fast build command is gone. You need the second half below.

Add 7.0 back under a second name

Install TypeScript 7 under a different alias so npx tsc resolves to it:

{
  "devDependencies": {
    // tooling reads this name → gets the stable 6.0 API
    "typescript": "npm:@typescript/typescript6@^6.0.0",
    // your build uses 7.0's fast tsc
    "typescript-7": "npm:typescript@rc"
  }
}

With both aliases in place: typescript-eslint and friends import typescript and get 6.0's API; npx tsc runs 7.0's fast compiler for your builds and editor integration. As the team put it, it's a textbook parallel change — 6.0 keeps linting tidy, 7.0 makes everything else faster.

The tsc6 binary

The compat package also gives you a tsc6 command. That's handy when you explicitly want to run the 6.0 compiler — for the side-by-side comparisons from earlier modules, or to double-check a result against the trusted baseline:

# run the 6.0 compiler explicitly
npx tsc6 --noEmit

# vs. 7.0 for the same project
npx tsc --noEmit

When does this go away?

This whole arrangement is temporary. It exists only because 7.0 lacks a stable programmatic API until 7.1. Once 7.1 ships that API and your tools (typescript-eslint, ts-morph, custom transformers) update to support it, you drop the aliases and let typescript simply mean the modern compiler. Treat the aliasing as a bridge, and revisit it when 7.1 lands.

Refresh your knowledge?

What the programmatic API is — Strada vs. Corsa. Beyond the command line, the compiler exposes a JavaScript programmatic API: functions to parse source into a syntax tree, ask about types, and walk the program. Tools build on it. typescript-eslint uses it to apply lint rules with type information; ts-morph uses it to read and rewrite code; custom transformers use it to modify output. The classic JS-based compiler's API is informally "Strada"; the native Go compiler's is "Corsa." These are different implementations, and the Corsa API isn't stable yet — which is the entire reason a 6.0-API compat package is needed during the transition.

What a linter and a formatter do. A linter (ESLint, via typescript-eslint for TS) analyses your code for problems and style violations — unused variables, unsafe patterns, rule breaches — and can often auto-fix them. A formatter (like Prettier) only rewrites whitespace and layout to a consistent style, without changing meaning. typescript-eslint is the bridge that lets ESLint understand TypeScript types, and to do that it needs the TypeScript API — which is why it, specifically, is the tool most affected by the package transition.

Why editor plugins are in the same boat. Many editor features for TypeScript are powered by the same programmatic API (or a server built on it). A plugin written against the Strada API won't automatically work against Corsa. That's why, during the preview, the editor experience comes through a dedicated native-preview extension rather than your existing plugins — a topic Module 5 picks up.

Goal

Set up one project where the build runs on TypeScript 7.0 but typescript-eslint still resolves the stable 6.0 API — and verify both work. This is the real coexistence pattern you'll use until 7.1.

Step 1 — A project with a linter

Start from a project that uses ESLint with typescript-eslint (or add it). The key dependency to notice is typescript-eslint, which peer-depends on typescript:

npm install -D eslint typescript-eslint

Step 2 — Wire the two aliases

Edit package.json so typescript points at the compat package and 7.0 lives under a second name:

{
  "devDependencies": {
    "eslint": "...",
    "typescript-eslint": "...",
    "typescript": "npm:@typescript/typescript6@^6.0.0",
    "typescript-7": "npm:typescript@rc"
  },
  "scripts": {
    "check": "tsc --noEmit",        // runs 7.0
    "lint": "eslint .",            // uses 6.0 API via the alias
    "check:6": "tsc6 --noEmit"       // explicit 6.0 baseline
  }
}

Then do a clean install so the aliases take effect:

rm -rf node_modules package-lock.json
npm install

Step 3 — Verify the build runs on 7.0

npx tsc --version    # expect a 7.0 version string
npm run check        # 7.0 type-check, fast

Step 4 — Verify the linter uses 6.0

npx tsc6 --version   # expect a 6.0 version string
npm run lint         # typescript-eslint resolves typescript → 6.0 API, works

If the lint run completes without "incompatible TypeScript version" or API errors, the coexistence is working: the linter found the 6.0 API it expected under the typescript name, while your build command ran 7.0.

Step 5 — Troubleshoot a duplicate-version error

If a dependency quietly pulls in a second copy of typescript, you can get contradictory errors. Diagnose and clean up:

# show every typescript copy in the tree and who asked for it
npm ls typescript

# then reinstall cleanly

Outcome. One project, two TypeScripts, no conflict: 7.0 for speed on builds and editor, 6.0's API for tooling that needs it. When 7.1 ships a stable API, remove the aliases and let typescript mean 7.x.
Refresh your knowledge?

What an npm alias actually does. The syntax "name": "npm:<real-package>@<range>" tells npm: "install real-package, but make it available in my project under name." So "typescript": "npm:@typescript/typescript6@^6.0.0" installs the compat package but lets every import ... from "typescript" resolve to it. Aliases are how you satisfy a tool that hard-codes a package name without being bound to whatever that name normally points to. The @rc in npm:typescript@rc is a dist-tag — a named pointer to the current release-candidate version, the same way @latest points at the newest stable.

What package-lock.json is for. Your package.json lists version ranges (like ^6.0.0). The lockfile (package-lock.json) records the exact versions and tree that were actually installed, so every machine and CI run reproduces an identical node_modules. When you change aliases, the lockfile can hold stale resolutions — which is why "delete node_modules and the lockfile, then reinstall" is the reliable reset. It forces npm to resolve the tree fresh against your new aliases.

Reading npm ls. npm ls <pkg> prints the dependency tree for that package — every copy present and which dependency required it. If you see two typescript entries at different versions, that's your duplicate: trace the parent that pulled the unwanted one, and either update it or dedupe. It's the fastest way to answer "why do I have two TypeScripts?"

Module 3

Breaking Changes & New Defaults

hard errors strict rootDir / types

This is the module where a clean 6.0 project sails through and a neglected one hits a wall of errors. TypeScript 7 doesn't change how type-checking works — but it does change the defaults you start from and turns 6.0's gentle deprecation warnings into hard stops. Open each idea.

hard errors Deprecations become hard errors +

TypeScript 6.0 introduced a set of deprecations — old flags and behaviours it warned you about but still allowed. TypeScript 7.0 removes that grace period: anything 6.0 deprecated now produces a hard error that stops the build. Dropped outright include the es5 compile target, node10 module resolution, the amd and umd module formats, and baseUrl path resolution.

The escape hatch you may have leaned on in 6.0 — "ignoreDeprecations": "6.0" — does not work in 7.0. It was always meant to buy time, not to be permanent. This is why the team's advice is to do the cleanup under 6.0 first, where the tooling and the warnings still help you, rather than meeting every error at once on the jump to 7.0.

new defaults 7.0 adopts 6.0's modern defaults +

Beyond removing old behaviour, 7.0 starts you from different out-of-the-box settings — the modern defaults introduced in 6.0. The headline ones: strict is on, module defaults to esnext, and target points at a current ECMAScript version (legacy es5 output is gone). These reflect a world where runtimes are evergreen and ESM is the norm.

If your tsconfig.json already sets these explicitly, nothing changes — your explicit values win. The projects that feel the shift are the ones that relied on the old implicit defaults without writing them down. The next tab walks each one.

type ordering stableTypeOrdering and the "honest build" test +

Internally, TypeScript numbers types in the order it encounters them, and that ordering can subtly affect things like the order of members in emitted declaration files. 6.0 and 7.0 can order differently. 6.0 offers a flag, --stableTypeOrdering, that makes its ordering match 7.0's — useful when you're comparing the two compilers' output and want to separate real differences from harmless ordering noise.

This gives a precise test for "am I ready?": a project that compiles cleanly under 6.0 with stableTypeOrdering enabled and ignoreDeprecations not set should produce identical results under 7.0. That's the "clean and honest build" — no deprecated flags sneaking by, ordering aligned. If your 6.0 build passes that bar, the jump to 7.0 is essentially a non-event. Note --stableTypeOrdering can slow 6.0 type-checking noticeably (up to ~25% on some codebases), so it's a migration-time check, not an everyday setting.

What a compiler flag and a default are. A compiler flag (also called a compiler option) is a setting that changes how the compiler behaves — for example whether it emits output, which JavaScript version it targets, or how strict it is. You can pass flags on the command line (--noEmit) or, more commonly, set them in tsconfig.json under compilerOptions. A default is the value a flag takes when you don't set it yourself. "Changing a default" means the compiler now assumes a different value than before when you stay silent — which is exactly why projects that never wrote a setting down can behave differently after an upgrade, while projects that set it explicitly are unaffected.

Warning vs. error. A warning flags something questionable but lets the build finish. An error stops the build. The whole story of this module is a set of things moving from the first category to the second: 6.0 warns, 7.0 errors. Warnings are easy to ignore, which is precisely why they don't carry over — 7.0 forces the issue.

How tsconfig settings resolve. Your effective configuration is built up in a predictable order: a base config you extends from, then your own compilerOptions overriding it, then any flags passed on the command line overriding that. The compiler's built-in defaults sit underneath everything, filling in only what nobody specified. The practical upshot for migration: the more you state explicitly in tsconfig.json, the less a defaults change can surprise you — explicit settings are immune to shifting defaults.

Accurate as of June 2026 The defaults and dropped flags below track the TypeScript 7.0 RC (inherited from 6.0). Preview details can shift — confirm against the official TypeScript 6.0 and 7.0 release notes before relying on a specific value.

The changed defaults at a glance

These are the out-of-the-box settings 7.0 adopts from 6.0. If your tsconfig.json already sets a value explicitly, that wins — this table is about what happens when you don't.

The team's own call-out: rootDir and types are the two most likely to surprise you — both are easy to fix once you know to look.

strict is on by default

If your project never wrote "strict": true, every type shortcut it quietly relied on for years is now flagged. This is the single most common source of upgrade errors. You have two honest paths: fix the errors (best), or set "strict": false explicitly while you work through them. What you should not do is pretend it isn't happening — the default has changed regardless.

// to opt back out temporarily while you migrate
{
  "compilerOptions": {
    "strict": false
  }
}

module: esnext — ESM by default

Output is now ES modules unless you say otherwise. For a browser project behind a bundler, that's usually fine. For a Node.js project that uses require() and CommonJS, you need to be explicit, or your emitted code won't run as expected:

// keep CommonJS for a Node project that needs it
{
  "compilerOptions": {
    "module": "commonjs",
    "moduleResolution": "node16"
  }
}
Note: node10 module resolution is one of the dropped behaviours — prefer node16, nodenext, or bundler.

types: [] — explicit type packages

Previously the compiler auto-loaded every @types/* package it found in node_modules into global scope. Now it loads none by default — you declare exactly which global type packages you depend on. If global names suddenly go "cannot find name," this is usually why:

{
  "compilerOptions": {
    // list the global @types you actually rely on
    "types": ["node"]
  }
}

rootDir — explicit output structure

rootDir used to be inferred from the common directory of your source files; now it defaults to the directory containing tsconfig.json. If your config sits at the project root and your code lives in ./src, your output folder structure can change unexpectedly (an extra src/ nesting under outDir). Set it explicitly to preserve the old layout:

{
  "compilerOptions": {
    +   "rootDir": "./src"
  },
  "include": ["./src"]
}

What strict actually turns on. "strict": true isn't a single check — it's an umbrella that enables a family of stricter type-checking options at once. The ones you'll meet most:

• strictNullChecks — null and undefined are no longer silently assignable to every type. You must handle the "it might be missing" case, which eliminates a huge class of "cannot read property of undefined" bugs.
• noImplicitAny — if the compiler can't infer a type and you didn't annotate one, it errors instead of silently falling back to any. This is the check that flags years of untyped shortcuts.
• strictFunctionTypes — stricter, safer checking of function-parameter compatibility.
• strictBindCallApply — bind, call, and apply are checked against the real signature.
• strictPropertyInitialization — class properties must be assigned in the constructor (or marked optional), so you don't read an uninitialised field.
• noImplicitThis and alwaysStrict round out the set.

Because these turn on together, flipping strict on a previously-loose codebase can surface many errors at once. That's not the compiler being difficult — each error is a real "this could be undefined" or "this is secretly any" that was there all along. You can also enable them one at a time (set strict: false and turn on individual options) to migrate gradually.

Goal

Take a deliberately "dishonest" TypeScript 6.0 project — one leaning on deprecated flags and implicit defaults — and drive it to a clean 7.0 build, fixing each hard error in turn. The exercise mirrors exactly what you'll do on a real codebase.

Step 1 — Start from a "dishonest" config

Here's a tsconfig.json that compiled (with warnings) under older TypeScript and breaks under 7.0. Each line in red is a problem 7.0 will reject:

{
  "compilerOptions": {
    "target": "es5",              // dropped
    "module": "amd",              // dropped
    "moduleResolution": "node10", // dropped
    "baseUrl": "./src",           // dropped
    "ignoreDeprecations": "6.0",  // no longer honoured
    "outDir": "./dist"
    // note: no strict, no rootDir, no types — all relying on old implicit defaults
  },
  "include": ["./src"]
}

Step 2 — Run 7.0 and read the errors

npx tsc --noEmit

You'll get a cluster of configuration errors (the dropped flags) plus a wave of type errors once strict kicks in by its new default. Don't panic at the count — work top-down: fix the config errors first, because some type errors disappear once the compiler is configured correctly.

Step 3 — Modernise the config

Rewrite the options to their supported equivalents, and make the previously-implicit settings explicit:

{
  "compilerOptions": {
    "target": "es2022",            // was es5
    "module": "nodenext",          // was amd
    "moduleResolution": "nodenext", // was node10
    "paths": { "@app/*": ["./src/*"] }, // replaces baseUrl
    "rootDir": "./src",            // make output structure explicit
    "types": ["node"],             // list globals you depend on
    "outDir": "./dist"
    // ignoreDeprecations removed entirely
  },
  "include": ["./src"]
}
Shortcut: the team ships a helper, ts5to6, that automates the most disruptive tsconfig changes (such as baseUrl removal and rootDir). Run it, review the diff, then finish by hand. Doing the cleanup under 6.0 first means you get its warnings and tooling to guide you.

Step 4 — Work through the strict errors

With the config clean, re-run and address the strict errors. Two common shapes:

// strictNullChecks: handle the missing case
function greet(name?: string) {
  return "Hi " + name.toUpperCase();        // name might be undefined
  return "Hi " + (name ?? "there").toUpperCase();
}

// noImplicitAny: annotate the parameter
function total(items) { /* ... */ }          // items is implicitly any
function total(items: number[]) { /* ... */ }

If the volume is large, you can stage it: set "strict": false and enable individual checks (strictNullChecks, then noImplicitAny, …) one at a time, landing each as its own change rather than all at once.

Step 5 — Confirm the "honest build"

Your target is a project that passes cleanly under 6.0 with --stableTypeOrdering on and no ignoreDeprecations. When that holds, 7.0 should agree:

# 6.0 clean + honest
node ./node_modules/typescript/bin/tsc --noEmit --stableTypeOrdering

# 7.0 should now pass with the same result
npx tsc --noEmit

When both pass, you've done the real work of the migration. Everything else in this course (packaging, editor, rollout) is logistics on top of a build that's already correct.

Refresh your knowledge?

How to read a TypeScript error message. A diagnostic has four parts: the file and position (src/app.ts:12:7), an error code (TS2532), a message, and often a chain of "related" notes underneath. Read it back-to-front: the deepest related note usually names the root cause, while the top line tells you where to fix it. The error code is searchable — pasting TS2532 into a search engine reliably finds an explanation. Don't be intimidated by a large count; errors cascade, so fixing one root cause (a wrong config option, a missing type package) often clears dozens of downstream messages at once.

The ignoreDeprecations escape hatch — and why it ends. In 6.0, adding "ignoreDeprecations": "6.0" to compilerOptions silences the deprecation warnings, letting a legacy config keep building while you plan the cleanup. It is explicitly a temporary measure. In 7.0 the deprecated behaviours are removed, not merely warned about, so the flag has nothing left to silence and stops working. The lesson the team repeats: use ignoreDeprecations to buy a little time under 6.0, but treat the underlying cleanup as the actual task — because 7.0 will require it regardless.

Why "do it under 6.0 first" is the recommended path. 6.0 is type-checking-compatible with 7.0 but still gives you warnings, the ignoreDeprecations safety valve, and migration tooling like ts5to6. That's a gentler environment to fix things in than 7.0, where the same issues are hard errors with no valve. Clean up on 6.0, confirm the honest build, then the version bump to 7.0 is anticlimactic — which is the goal.

Module 1

Why TypeScript 7 Exists

native compiler Project Corsa Go port

Most TypeScript releases add or refine type-system features. TypeScript 7 is different in kind: the language barely moves, while the thing that runs it was replaced. Three ideas explain what that means — open each one.

native compiler A new engine, not a new dialect +

When you adopt TypeScript 7, you are adopting a faster engine for the same language. The team rebuilt the compiler and tooling so they run as native code rather than as JavaScript, and the result is often about 10× faster than TypeScript 6.0, with substantially lower memory use.

This speed is not only a CI win. The same engine powers your editor: every hover, every "go to definition," every red squiggle is the type-checker working. On a large codebase that previously meant slow editor startup and laggy feedback. Native speed and shared-memory parallelism improve the command-line build and the editing experience at once — which is why teams piloting it reported both faster CI and a more fluid editor. How that speed is delivered is the subject of Module 2.

Project Corsa A port, not a rewrite +

"Project Corsa" is the name of the year-long effort to move the compiler to its new foundation. The most important word is port, not rewrite. The team did not reimagine how type-checking should work; they methodically moved the existing implementation across, keeping the type-checking logic structurally the same.

That choice is the through-line of this whole course. Because the checker was ported rather than reinvented, a program that type-checks cleanly under 6.0 produces the same results under 7.0. As a measure of how complete this is: of roughly 20,000 compiler test cases, about 6,000 produce at least one error under 6.0, and 7.0 reproduces an error in all but a small handful — and those exceptions are known incomplete work or deliberate, documented changes. The errors, inferences, and hover types are intended to be identical. What changes is how fast you get them, plus a set of default settings covered in Module 3.

Go port Why Go, and not Rust or WebAssembly +

The new compiler is written in Go. It is reasonable to ask why Go, rather than Rust or a WebAssembly build. The honest answer the team gave was pragmatic: Go's concurrency model and memory layout lined up well with how the existing compiler was already structured, which made a faithful port feasible in about a year.

A ground-up rewrite in another language would have taken far longer and risked drifting away from the exact semantics teams depend on. Speed of porting — not squeezing out the last possible nanosecond — drove the decision. The pay-off is a compiler that is dramatically faster while still behaving like the one you already know.

Refresh your knowledge?

What does a TypeScript compiler actually do? If "the compiler was ported" is to mean anything, it helps to recall the compiler's job. It runs in three stages. Parse: it reads your .ts source and builds a structured tree (an abstract syntax tree) representing the code's shape. Type-check: it walks that tree, infers types you didn't write, and verifies the types you did — this is the stage kept structurally identical in the Go port. Emit: it strips the types away and writes plain JavaScript (types do not exist at runtime). The TypeScript 7 claim is precise in these terms: the type-check stage behaves the same, emit produces the same JavaScript, and the port changed only how fast all three run.

Why is TypeScript considered a robust language? A short refresher on the features that earn that reputation:

Static type safety. TypeScript checks types at compile time, before your code ever runs. A function that expects a number cannot silently be handed a string; the mismatch is caught at your desk, not in production. This is the central value-add over plain JavaScript, and it is exactly what the native compiler makes faster without changing.

Type inference. You rarely have to annotate everything. Write const total = 3 + 4 and TypeScript infers total: number on its own. Inference keeps the safety while keeping the code close to the JavaScript you'd write anyway.

Interfaces and structural typing. TypeScript types describe the shape of data. Two objects are compatible if their shapes match, regardless of where they came from — "if it has the right properties, it fits." This structural approach models real-world JavaScript objects naturally.

Generics. Reusable code that preserves type information. Array<string>, Promise<User>, or your own function first<T>(items: T[]): T all keep the specific type flowing through, instead of collapsing to any.

Union, literal, and narrowing. A value can be "open" | "closed" and nothing else; control-flow analysis then narrows the type as you check it (inside if (x === "open"), TypeScript knows x is "open"). This catches whole categories of bugs that plain JavaScript would let through.

Iterators and iterables. TypeScript models the iteration protocol — anything with a [Symbol.iterator]() method that yields values — so for…of, spread [...x], and destructuring all type-check correctly across arrays, Map, Set, generators, and your own iterable classes. Generator functions (function*) are typed as Generator<T>/Iterator<T>, and downlevel iteration is handled at emit. Note for this course: iterators are an established TypeScript feature, not new in 7.0 — the native compiler type-checks them exactly as 6.0 did. They're included here as part of what makes the language robust, not as a 7.0 feature.

Enums, tuples, and utility types. Rounding out the toolkit: enum for named constant sets, fixed-length typed tuples, and built-in utility types like Partial<T>, Pick<T,K>, and Readonly<T> that transform existing types without rewriting them. Together these let you describe intent precisely and let the compiler hold you to it.

Every one of these features behaves identically under TypeScript 7. That is the point of a port: the robustness you rely on is unchanged — you simply get the compiler's answer roughly ten times faster.

Before you install: Node.js

The native compiler is distributed as an npm package with prebuilt platform binaries, so you need Node.js and npm available. The package requires Node.js 16.20.0 or newer; any current LTS release (Node 20 or 22) is a safe choice. Check what you have:

# confirm Node and npm are present
node -v   # should print v16.20.0 or higher
npm -v
Note. This is unrelated to Node's own "run .ts files directly" feature (type stripping). That feature only erases types at runtime and does no type-checking. The TypeScript 7 compiler is what actually checks your types — the two are complementary, not the same thing.

Install the compiler

During the RC period the compiler ships under the preview package name @typescript/native-preview. The separate package name is deliberate: it means the preview cannot collide with a typescript install you already have. Install it as a dev dependency in your project:

# per-project (recommended)
npm install -D @typescript/native-preview

Or globally, if you want the command available everywhere:

# global
npm install -g @typescript/native-preview
The binary name depends on the build. For the TypeScript 7.0 RC and later, the command is tsc — the same name you already know. For earlier native-preview builds (the betas and nightlies), the command was tsgo. If tsc isn't found after installing a preview build, try tsgo — it's the same compiler under the older name. At the stable 7.0 release the compiler moves into the regular typescript package and the @typescript/native-preview name retires.

Verify it works

Confirm the install by printing the version. Using npx runs the project-local copy without needing a global install:

# RC and later
npx tsc --version

# earlier preview builds used the tsgo name
npx tsgo --version

For the editor, install the TypeScript Native Preview extension from the VS Code Marketplace. It updates frequently and brings the same native speed to hovers, go-to-definition, and error reporting. (We cover the editor story in depth in Module 5.)

Refresh your knowledge?

Node.js, npm, and npx — what each one is. Node.js is the runtime that executes JavaScript outside a browser; it's what runs the compiler's binary on your machine. npm is the package manager bundled with Node — it installs packages listed in your project and records them in package.json. npx runs a command from a package without a separate global install, preferring the project-local copy in node_modules/.bin. That's why npx tsc reliably runs this project's compiler rather than whatever happens to be installed globally.

Global vs. dev install. npm install -g puts a package on your whole machine (handy for a CLI you use everywhere); npm install -D (short for --save-dev) puts it in the current project as a development dependency, recorded under devDependencies in package.json. Per-project installs are usually preferable: they pin the exact version your project expects, so teammates and CI all build with the same compiler.

Managing Node versions. The native compiler needs Node 16.20.0 or newer. If you juggle projects on different Node versions, a version manager such as nvm lets you switch: nvm install 22 then nvm use 22. Checking node -v before installing saves a class of confusing errors that are really just an out-of-date runtime.

Node's own TypeScript switches — and why they aren't a type-checker. Recent Node versions can run .ts files directly via type stripping: they erase the type annotations and execute the remaining JavaScript. Related runtime flags you may encounter: --experimental-strip-types (erase types and run; default-on in current Node), --experimental-transform-types (also transform non-erasable syntax like enum and parameter properties), and --no-experimental-strip-types (turn the behaviour off). The crucial point: type stripping does not check your types — it removes them. It will happily run code that has type errors. The TypeScript 7 compiler is what actually validates types. The two are complementary: Node's stripping is about executing TypeScript quickly; the compiler is about verifying it. This course is about the compiler.

Type-check a project

The native compiler reads your existing tsconfig.json unchanged — no configuration changes are needed to try it. The most common first command type-checks without writing any output, which is the fastest way to feel the speed difference:

# type-check only, no emit — reads ./tsconfig.json
npx tsc --noEmit

# point at a specific project file
npx tsc -p tsconfig.json --noEmit

The commands mirror what you already run with TypeScript 6.0. That is intentional: the entry point and the flags are the same, so muscle memory carries over.

Build (emit JavaScript)

Drop --noEmit to produce JavaScript, exactly as before:

# compile to JS using your tsconfig outDir
npx tsc

# watch mode — recompile on save
npx tsc -w
RC feature gaps to know. The native compiler is type-checking-complete but a few build-side features are still landing. During the preview, declaration-file emit (--declaration / .d.ts), some downlevel emit targets, and full --build (project-references orchestration) may be incomplete. The practical pattern teams use: type-check with the native compiler for speed, and keep TypeScript 6.0 (tsc from the typescript package) for any production emit that touches those gaps until the stable release. Module 4 covers this coexistence in detail.

Wire it into package.json

A clean way to adopt incrementally is to add native-compiler scripts alongside your existing ones, so you can compare both on demand:

// package.json
{
  "scripts": {
    "check": "tsc --noEmit",          // native: fast type-check
    "check:legacy": "tsc-6 --noEmit",  // fallback (see Side-by-side tab)
    "build": "tsc"
  }
}

Because the native compiler honours all your existing tsconfig.json options and project references with no config changes, the only thing that changes in day-to-day use is the speed.

Refresh your knowledge?

package.json — the project's manifest. This file at the project root describes your project to npm: its name and version, its dependencies, and its scripts. The scripts block is what you run with npm run <name> — for example a "check": "tsc --noEmit" entry lets the whole team type-check the same way with one command. Dependencies split into dependencies (needed at runtime in production) and devDependencies (needed only while developing or building — the compiler belongs here, since types are erased before production runs).

tsconfig.json — the compiler's settings. This file tells the TypeScript compiler how to treat your project: which files to include, what to emit, and how strict to be. The native compiler reads it unchanged. The options most relevant to this module:

• noEmit — type-check only, write no output (the fast feedback loop).
• outDir — where compiled JavaScript is written.
• target — the JavaScript version to emit (e.g. es2022).
• module — the module system for output (e.g. esnext, nodenext).
• strict — turns on the full family of strict type-checking checks (covered in Module 3).
• declaration — also emit .d.ts type-definition files (one of the build features still being completed in the preview).
• references — lists other TypeScript projects this one depends on, for monorepo/project-reference builds (the subject of Module 2's --builders).

How npx finds the right binary. When you run npx tsc, npx looks first in the project's node_modules/.bin — so it runs the compiler version your package.json pinned, not a global one. This is why committing your dev dependency and using npx (or an npm script) gives every machine the same compiler. Running a bare tsc with no local install would fall back to whatever is global, which is the usual cause of "works on my machine" version drift.

Why run both at once?

Because the native compiler ships under the separate @typescript/native-preview package, it can live in the same project as your existing typescript install without conflict. That is exactly what you want during adoption: the native compiler for fast type-checks, the trusted 6.0 compiler for anything still being ported (declaration emit, full --build), and a way to compare the two side by side to confirm they agree.

The team has explicitly committed to keeping the two installable together "without any conflicts around which tsc is which" for the foreseeable future, so this is a supported workflow, not a hack.

Install both

# the classic compiler (TypeScript 6.0) …
npm install -D typescript

# … and the native preview alongside it
npm install -D @typescript/native-preview

Now resolve the "which tsc?" question. The classic package's binary is tsc; the native RC binary is also tsc. To call them unambiguously in scripts, reference them by package path rather than the bare name:

// package.json — explicit, unambiguous
{
  "scripts": {
    // native preview (fast type-check)
    "check": "tsgo --noEmit",
    // classic 6.0 compiler, by its package's bin path
    "check:6": "node ./node_modules/typescript/bin/tsc --noEmit",
    // run both and compare — they should agree
    "check:both": "npm run check && npm run check:6"
  }
}
Tip. On RC builds where the native binary is tsc, the tsgo alias is still provided too, which is the simplest way to keep the two names distinct in scripts: tsgo = native, tsc (from the typescript package's bin) = classic.

The compatibility package for tooling

There is one more piece. Many tools in your stack — linters such as typescript-eslint, formatters, and editor plugins — import TypeScript's programmatic API directly. The native compiler's programmatic API is not stable yet; it is not expected until TypeScript 7.1 or later, several months after the 7.0 release. Until then, tools that depend on the 6.0-era API need that API to remain available.

For this, the team publishes a compatibility package, @typescript/typescript6. The pattern: run the native compiler for your own type-checks and builds, while tooling that needs the older API resolves it through the compat package. This keeps your linter and editor plugins working during the transition.

# API-compat shim for tools that import the 6.0 API
npm install -D @typescript/typescript6
Heads-up. If a dependency quietly pulls in an older typescript version, you can get confusing duplicate-version errors. If that happens, find the offending package (inspect package-lock.json), and reinstall cleanly. Module 4 goes deeper on the packaging and tooling transition.
Refresh your knowledge?

What "TypeScript 6.0" means here, and why it's the reference point. Throughout this course, 6.0 is the baseline the native compiler is measured against — both for speed ("about 10× faster than 6.0") and for correctness ("produces the same results as 6.0"). When you run the two side by side, 6.0 is the trusted answer key. Knowing a few things about it makes the comparison meaningful.

The typescript package and the tsc binary. The classic compiler is published on npm as typescript, and its command-line tool is tsc. This is the compiler your project almost certainly uses today. The native preview deliberately ships under a different package name precisely so it can sit beside this one — at stable release, the native compiler takes over the typescript package and the tsc name.

The programmatic API (Strada) and why tooling depends on it. Beyond the command line, the classic compiler exposes a JavaScript API that other tools call into — to parse code, inspect types, and walk the syntax tree. Linters (typescript-eslint), formatters, and many editor plugins are built on this API. The native compiler's codebase is informally called "Corsa," and the classic JS-based one "Strada." The native programmatic API is still being designed and won't be stable until 7.1 or later — which is the entire reason the @typescript/typescript6 compatibility package exists: it keeps the 6.0-era API available for tools that need it while you adopt the native compiler for type-checking.

6.0's deprecations and changed defaults. TypeScript 6.0 introduced a set of deprecations and new default settings — for example moving away from older module-resolution modes. These are gentle warnings in 6.0, but TypeScript 7 promotes many of them to hard errors and adopts the new defaults outright. That's why a project that's clean and warning-free under 6.0 transitions most smoothly. The full list and how to reconcile it is the subject of Module 3.
